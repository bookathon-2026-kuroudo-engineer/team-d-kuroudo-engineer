# 読み手のペルソナ

- 30歳、大卒 工学部建築学科
- macを仕事で使っている。趣味はWindows
- お仕事: 社内でインフラ工事の図面を引く業務
- 社内システムの図面管理システムでとても困っている
- 大学の時は、都市工学の研究をしていたので、excelのマクロとかで公共データの分析をしていた。pythonも少しだけ書いたことあるけど、がっつりしたチーム開発の経験はない。研究でちょっと分析に使ったくらい。

# 章立て案

## 第I部　現場の知識を改善につなげる

### 第1章　ソフトウェア改善はエンジニアだけの仕事ではない

- 日常的に使う人が持っている知識
- エンジニアには見えにくい業務上の問題
- OSSと社内システムに共通する構造
- コードを書かない貢献
- AIエージェントによって参加のハードルが下がったこと
- AIを使っても、人間が判断すべきこと

「OSSへの貢献」という説明を、「ソフトウェア改善への参加」に置き換えます。

### 第2章　困りごとを変更要求に変える

- 「使いにくい」を具体化する
- 現在の挙動
- 期待する挙動
- 業務上なぜ問題なのか
- 影響を受ける人
- 発生頻度
- 例外条件
- 完了条件

社内システムでは、単なるバグだけでなく「業務ルールと実装が合っていない」というケースが多くなります。

### 第3章　問題を共有し、着手してよいか確認する

- Issueやチケットを書く
- 社内チャットで相談する
- 開発チームや管理者を特定する
- 既存の要望や計画を確認する
- 実装前に相談すべき変更
- 自分で直してよい範囲
- 承認が必要な範囲

OSSではIssue、社内ではJira、Backlog、GitHub Issues、Slackなどを想定します。

## 第II部　AIとシステムを理解する

### 第4章　そのソフトウェアがどのように管理されているかを知る

- ソースコードの所在
- ドキュメント
- Issue管理
- 開発環境
- テスト環境
- 本番環境
- 管理責任者
- 変更の承認フロー
- OSSにおけるCONTRIBUTING
- 社内における開発ルール

社内システムでは、リポジトリにアクセスできるからといって変更してよいとは限りません。

### 第5章　安全な作業環境を準備する

- 本番環境を直接触らない
- 開発環境、検証環境、本番環境
- リポジトリを取得する
- 作業用ブランチを作る
- ローカルで起動する
- テストデータを用意する
- 個人情報や機密情報を使わない
- AIに渡してよい情報を確認する

OSS版では開発環境構築が中心でしたが、社内版では環境分離と情報管理が重要になります。

### 第6章　AIエージェントにシステムを調査させる

- 最初から修正させない
- 関係する画面、処理、ファイルを探す
- データの入力から出力までをたどる
- 業務ルールがどこに実装されているか調べる
- 類似機能を探す
- 過去の変更を調べる
- 調査結果に根拠を付けさせる
- 不明な点は担当者へ質問する

社内ではコードだけでなく、設定ファイル、データベース、外部サービス、運用手順が挙動に関係する場合があります。

## 第III部　小さな改善を作る

### 第7章　変更を小さく設計する

- 一つの問題に絞る
- 変更範囲を限定する
- 業務への影響範囲を考える
- 既存の運用を壊さない
- コード変更以外の解決策を検討する
- 設定変更、手順変更、文言修正との比較
- 元に戻せる変更にする

社内システムでは、「コードを直すこと」が必ずしも最適解ではありません。

例えば次の選択肢があります。

- マニュアルを直す
- 入力チェックを追加する
- 初期値を変更する
- 設定を変更する
- データを補正する
- 業務フローを変更する
- コードを変更する

### 第8章　AIに変更案を作らせる

- 目的、制約、完了条件を渡す
- 調査と実装を分ける
- 変更計画を先に作らせる
- 変更してはいけない部分を伝える
- 小さな差分を作らせる
- 差分を説明させる
- 不明点を推測させない
- 自分で説明できない変更を採用しない

### 第9章　動作確認と受け入れ確認

- ビルドが通ること
- テストが通ること
- 業務上正しく動くこと
- 修正前後を比較する
- 正常系、異常系、境界条件
- 既存データへの影響
- 他部署や他ユーザへの影響
- 操作手順やマニュアルへの影響
- 現場の利用者による受け入れ確認

OSSの場合はPRを提出しますが、社内では「受け入れ確認」が特に重要になります。

## 第IV部　安全に変更を届ける

### 第10章　AIが作った変更をレビューする

- 要求と関係のない変更
- データ削除
- 権限変更
- 個人情報や機密情報の扱い
- 外部通信
- ログへの情報出力
- 依存ライブラリの追加
- パフォーマンスへの影響
- 障害時の挙動
- 元に戻せるか
- 自分だけでは判断できない変更

### 第11章　エンジニアや管理者に変更を渡す

- Pull Request
- 社内レビュー
- 変更申請
- リリース依頼
- 問題、変更内容、確認結果を書く
- 業務上の背景を説明する
- 技術的な判断をエンジニアに委ねる
- レビューコメントへ対応する
- 自分でマージやリリースをしてよいか確認する

ここでは「メンテナとのコミュニケーション」を、より広い「責任者との協働」に置き換えます。

### 第12章　継続的に改善へ参加する

- 問題報告
- 仕様整理
- 受け入れテスト
- ドキュメント
- データ整備
- 小さな自動化
- ログや利用状況の分析
- 定期的な改善会
- ドメインエキスパートとエンジニアの役割分担
- 自分が詳しい領域を持つ
- 必要な技術を少しずつ学ぶ
