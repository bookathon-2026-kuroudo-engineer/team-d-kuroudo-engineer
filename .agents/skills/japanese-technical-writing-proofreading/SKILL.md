# 日本語技術文書の校正・推敲

## 目的

与えられた文章を、読者が余計な読解作業をせず、本文だけでも大意を追える文章に整える。

この skill は、誤字脱字だけでなく、次の3階層を確認する。

1. **文**：語順、読点、用語説明、指示語、冗長表現
2. **段落**：1段落1トピック、トピックセンテンス、段落の粒度
3. **アウトライン**：見出しの階層、ストーリー、命名規則

## 必須入力

校正を開始する前に、次の情報がすべて与えられていることを確認する。

- 校正対象の全文
- 対象読者
- 文章の目的
- 読後に読者へ期待する理解または行動
- 媒体（書籍、雑誌、ブログ、提案書、README、スライド原稿など）
- 文体や表記の指定。指定がない場合は「指定なし」という回答
- 文字数・分量の制約。制約がない場合は「制約なし」という回答
- 校正の範囲（誤字脱字のみ、表現改善、構成変更、全面的な書き換えなど）
- 技術用語、固有名詞、表記、主張のうち、変更してはいけないもの。ない場合は「なし」という回答

## 情報不足時の必須動作

上記のうち1項目でも不足している場合は、**校正・推敲・書き換えを開始してはならない**。本文から推定したり、一般的な前提で補ったりせず、不足している情報を利用者へ必ず質問する。

質問するときは、次の規則に従う。

1. すでに与えられている情報は聞き直さない。
2. 不足項目をまとめて、回答しやすい形で列挙する。
3. 「指定なし」「制約なし」「なし」と回答できる項目は、その選択肢を明示する。
4. 利用者の回答を受け取るまでは、校正案、部分的な修正版、推定に基づく提案を出さない。
5. 回答後も不足が残る場合は、残りの情報を再度要求する。

質問テンプレート：

```markdown
校正を始めるため、次の情報を教えてください。

- 対象読者：
- 文章の目的：
- 読後に期待する理解・行動：
- 媒体：
- 文体・表記の指定（なければ「指定なし」）：
- 文字数・分量の制約（なければ「制約なし」）：
- 許可する修正範囲：
- 変更してはいけない用語・表記・主張（なければ「なし」）：
```

## 基本方針

- 著者の意図、技術的意味、主張の強さを勝手に変えない。
- 正しさを確認できない事実や技術説明を創作しない。不明点は指摘として残す。
- コード、識別子、コマンド、URL、引用文、製品名、固有名詞は、明白な誤りでない限り変更しない。
- すべてを機械的に削るのではなく、読みやすさに寄与する表現は残す。
- 個別の文だけでなく、前後関係と文書全体の構造を見て判断する。
- 変更量よりも、読者が迷わないことを優先する。

## 実行手順

1. 必須入力がすべて揃っているか確認する。
2. 不足があれば、校正を開始せず、「情報不足時の必須動作」に従って利用者へ質問する。
3. 必須入力がすべて揃ったことを確認してから、対象読者、目的、媒体、制約、修正範囲、変更禁止事項を作業条件として固定する。
4. 見出しだけを順に読み、アウトラインを確認する。
5. 各段落の第1文だけを読み、本文の要約として成立するか確認する。
6. 各段落が1トピックになっているか確認する。
7. 各文を、以下のチェックリストに従って確認する。
8. 修正版を作成する。
9. 修正によって技術的意味、因果関係、主張の強さが変わっていないか再確認する。
10. 修正結果と、判断を要する指摘を分けて出力する。

# 校正チェックリスト

## A. 文

### A01. 長い修飾語を先に置く

- [ ] 同じ語に複数の修飾語が係る場合、原則として長い修飾語を前、短い修飾語を後に置いているか。
- [ ] 修飾語が誤った語に係って読める並びになっていないか。

例：

- 修正前：白い横線の引かれた厚手の紙
- 修正後：横線の引かれた厚手の白い紙

### A02. 通常の語順を崩した箇所に読点を置く

- [ ] 主語の強調などで通常の語順を変更した場合、係り受けの誤読を防ぐ読点があるか。
- [ ] 読点がないため、主語や修飾語の係り先を誤解できる文になっていないか。

例：

- 修正前：渡辺刑事は血まみれになって逃げ出した賊を追いかけた。
- 修正後：渡辺刑事は、血まみれになって逃げ出した賊を追いかけた。

### A03. 長い修飾語どうしの境界に読点を置く

- [ ] 長い修飾語が2つ以上並ぶ場合、その境界が読点などで明示されているか。
- [ ] 読者が途中まで読まないと文の構造を判断できない状態になっていないか。

### A04. 重要度の低い用語にも短い説明を添える

- [ ] 本筋ではない専門用語でも、読者が読み進めるために必要な最低限の説明があるか。
- [ ] 用語名だけを突然提示していないか。
- [ ] 「何をするものか」「どの種別か」を一言で補える箇所がないか。

例：

- `Foo` → `ビルドを実行するFooメソッド`
- `Bar` → `Barという最適化手法`

### A05. 用語説明の範囲を対象読者・目的・分量に合わせる

- [ ] どの基礎概念まで説明するかが、対象読者と文書の目的に合っているか。
- [ ] 似た難易度の用語について、説明するものとしないものが不規則に混在していないか。
- [ ] 分量に余裕があり線引きが曖昧な場合、短い説明を一貫して付けるほうが読みやすくならないか。

### A06. 用語の種別を明記する

- [ ] メソッド、クラス、インタフェース、型、配列、要素、属性など、対象の種別が明示されているか。
- [ ] 名称だけでは、読者がそれを何として扱うべきかわからない箇所がないか。

例：

- `h1` → `h1要素`
- `h1要素` → `見出しを表すh1要素`

### A07. 指示代名詞を多用しない

- [ ] 「これ」「それ」「このこと」などの指示先が一意か。
- [ ] 指示語が連続し、何を指すのか追跡しにくくなっていないか。
- [ ] 曖昧な「これは」を、「このメソッドは」「この設定は」など具体的な名詞に置き換えられないか。

### A08. 冗長表現を簡略化する

- [ ] 意味を変えずに削れる「しまう／しまい／しまっ」がないか。
- [ ] 不要な「ような／ように」「という／といった」がないか。
- [ ] 不要な「てみる／てみて」がないか。
- [ ] 「非常に」「とても」「たいへん」「大変」などの強調語を多用していないか。
- [ ] 削除するとニュアンスや意図が変わる場合は残しているか。

例：

- `情報が失われてしまう` → `情報が失われる`
- `表1に示すようなツール` → `表1に示すツール`
- `試してみてください` → `試してください`

## B. 段落

### B01. 1段落で1トピックだけを扱う

- [ ] 意味のまとまりごとに段落が分かれているか。
- [ ] 1つの段落に、独立して説明すべき複数の話題が混在していないか。
- [ ] 段落を分けた場合、各段落の関係がわかる接続になっているか。

### B02. 段落の第1文をトピックセンテンスにする

- [ ] 各段落の第1文が、その段落の主題や結論を説明しているか。
- [ ] 第1文が概要、続く文が根拠・詳細・例という構造になっているか。
- [ ] 各段落の第1文だけを読んでも、文章全体の要約として大意を追えるか。

### B03. 段落の粒度を見出しの粒度に合わせる

- [ ] 大きな見出しの直下では段落が過度に細切れになっていないか。
- [ ] 細かく分けた見出しの直下で、1段落が複数の小見出し相当の内容を抱えていないか。
- [ ] 見出しを細分化した結果、各段落のトピックも適切に小さくなっているか。

### B04. 本文だけでも大意を追えるようにする

- [ ] 見出し、ソースコード、図、表を読み飛ばしても、本文だけで話の流れがわかるか。
- [ ] 図表やコードにだけ登場した用語を、本文で説明なしに使っていないか。
- [ ] 図表やコードを参照する前に、それが何を示すかを本文で説明しているか。

## C. アウトラインと見出し

### C01. 見出しの階層と数を媒体に合わせる

- [ ] 見出しが少なすぎて長い塊になっていないか。
- [ ] 見出しが多すぎて文章が細切れになっていないか。
- [ ] 紙面、画面幅、記事の長さ、読まれ方に合った階層数になっているか。
- [ ] 媒体固有のルールがある場合、それを優先しているか。

目安は絶対的な規則ではない。媒体に応じて調整する。

### C02. 見出しだけでストーリーを追えるようにする

- [ ] 最上位の見出しだけを読んでも、文書全体の流れがわかるか。
- [ ] 上位と中位の見出しを読めば、より具体的な流れがわかるか。
- [ ] すべての見出しを読めば、本文の論理展開を把握できるか。
- [ ] 深い階層へ進むほど詳細が増える構造になっているか。

### C03. 見出し単体で意味がわかるようにする

- [ ] 名詞だけを置いた曖昧な見出しになっていないか。
- [ ] 必要に応じて、その対象の役割・目的・特徴まで見出しに含めているか。
- [ ] 文脈から切り離しても、何を説明する節か判断できるか。

例：

- `プロダクトオーナー` → `プロダクトオーナー ── プロダクトの最終責任者`

### C04. 問いを掲げた見出しには、配下の見出しで答えを示す

- [ ] 「○○とは何か」「○○の特徴」などの見出しに対し、配下の見出しだけでも答えがわかるか。
- [ ] 抽象的な問いの下に、単なる項目名だけが並んでいないか。
- [ ] 各小見出しが、上位見出しへの回答になっているか。

### C05. 読者に分類を推測させない

- [ ] 同じ階層に材料、手順、原因、結果など異なる種別の項目を混在させていないか。
- [ ] `材料1`、`材料2`、`手順1`のような番号だけで関係を示していないか。
- [ ] 「材料」「手順」などの中間見出しを設け、構造を明示できないか。

### C06. 下位階層ほど見出しを多くする

- [ ] 概要から詳細へ進むにつれて、見出しの数が自然に増えているか。
- [ ] 上位階層より下位階層のほうが極端に少なく、詳細化されていない構造になっていないか。
- [ ] 例外がある場合、その構造に明確な理由があるか。

これは傾向を確認するためのヒューリスティックであり、数だけを機械的に合わせない。

### C07. 下位階層ほど見出しを具体的かつ長くする

- [ ] 上位見出しは概要を表し、下位見出しは対象・条件・結果をより具体的に示しているか。
- [ ] 下位見出しが上位見出しと同程度に抽象的なままになっていないか。
- [ ] 詳細を表すために必要な情報を省略しすぎていないか。

### C08. 見出し末尾の命名規則を統一する

- [ ] 体言止めと用言止めの使い分けに一貫した規則があるか。
- [ ] 同じ階層・同じ種類の見出しで、末尾表現が不規則に揺れていないか。
- [ ] 原則を体言止めにする場合、手順など用言止めを使う箇所が定義されているか。
- [ ] 意味の薄い「○○について」を避けられないか。

例：

- `Webサーバの導入について` → `Webサーバの導入`

### C09. 「AとB」だけの見出しを避ける

- [ ] 「AとB」が単なる列挙にとどまり、両者の関係や節の目的が不明になっていないか。
- [ ] 比較、関係、違い、連携、見直しなど、何を説明するかまで見出しに含められないか。

例：

- `XPとパターン` → `XPとパターンの関係`
- `利用者と設計者` → `利用者と設計者の関係を見直す`

# 修正の優先順位

次の順で修正する。

1. 誤読や意味の取り違えを招く問題
2. 技術用語や指示先が不明な問題
3. 段落や見出しの構造上の問題
4. 冗長表現や表記の揺れ
5. 好みの範囲にとどまる文体上の変更

上位の問題を直すために大きな書き換えが必要な場合は、変更理由を明示する。

# 出力形式

必須入力が揃っていない場合は、以下の校正結果を出力せず、不足情報を求める質問だけを出力する。必須入力が揃っている場合のみ、以下の順で出力する。

## 1. 校正後の文章

全文を書き換える。変更していない箇所も含め、利用者がそのまま使える形にする。

## 2. 主な修正点

重要な修正だけを、次の形式で示す。

| ID | 修正前 | 修正後 | 理由 |
|---|---|---|---|
| A07 | これは〜 | このメソッドは〜 | 指示先を明確化 |

軽微な表記修正をすべて列挙する必要はない。

## 3. 要確認事項

技術的事実、対象読者、著者の意図など、AIだけでは確定できない点を列挙する。確定できない内容を修正版へ勝手に反映しない。

## 4. チェック結果

各チェック項目を次のいずれかで簡潔に報告する。

- `PASS`：問題なし
- `FIXED`：修正済み
- `REVIEW`：著者判断が必要
- `N/A`：該当なし

# 最終確認

出力前に必ず確認する。

- [ ] 技術的意味を変えていない
- [ ] 主張の強さを変えていない
- [ ] 新しい事実を創作していない
- [ ] 指示語の参照先が明確である
- [ ] 各段落が1トピックである
- [ ] 各段落の第1文で主題がわかる
- [ ] 本文だけでも大意を追える
- [ ] 見出しだけでもストーリーを追える
- [ ] 見出しの命名規則が一貫している
- [ ] 修正版をそのまま利用できる形式で出力している
