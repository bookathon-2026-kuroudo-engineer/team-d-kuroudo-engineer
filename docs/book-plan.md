# 書籍企画書（改訂版）

## 本のテーマ

### 従来案

> ドメイン知識を持つ非エンジニアが、AIエージェントを使ってOSSに貢献する

### 変更案

> 業務や製品の知識を持つ非エンジニアが、AIエージェントを使ってソフトウェア改善に参加する

ここでいう「貢献」はコード提出だけではない。

- 問題の発見と再現
- 要求や仕様の整理
- UIや業務フローの改善提案
- テストケースの作成
- ドキュメントやヘルプの修正
- 設定、データ、テンプレートの変更
- 小さなコード修正
- エンジニアが作った変更の受け入れ確認
- AIが作った修正案の検証

この言い換えにより、読者の入口を大きく広げられる。

## 本書で扱う二つのケース

本全体を、次の二つに共通する方法論として書く。

### 1. 公開プロジェクトへの貢献

- OSS
- GitHub上の公開プロジェクト
- コミュニティ運営のゲームやツール
- プラグインや拡張機能

### 2. 組織内のソフトウェアへの貢献

- 社内業務ツール
- 社内ポータル
- 営業、経理、人事などの業務システム
- 自社で運営するWebサービス
- 社内向けスクリプトや自動化
- 開発チームが管理する設定やテンプレート

### 共通する本質（改善の流れ）

1. 現場で問題を見つける
2. 正しい挙動を定義する
3. AIにシステムを調査させる
4. 変更案を作る
5. 安全性を確認する
6. 適切な責任者にレビューしてもらう

## タイトルの方向性

OSSという言葉を外す場合の候補：

- AIエージェントと始めるソフトウェア改善
- コードを書けなくても、ソフトウェアは改善できる
- 現場の知識をソフトウェアに反映する技術
- 非エンジニアのためのAI駆動ソフトウェア改善
- 使う人が直すソフトウェア
- ドメイン知識をコードに変える
- AIと参加するソフトウェア開発
- 現場から始めるソフトウェア改善

内容を最も正確に表す第一候補：

> **現場の知識をソフトウェアに反映する――AIエージェントと始める開発参加**

「開発」ではなく「開発参加」とすることで、エンジニアになるための本ではないことを表す。

## 改訂後の章立て（全12章・4部構成）

二日で書く前提で、前回の12章構成を次のように修正する。

### 第I部　現場の知識を改善につなげる

#### 第1章　ソフトウェア改善はエンジニアだけの仕事ではない

- 日常的に使う人が持っている知識
- エンジニアには見えにくい業務上の問題
- OSSと社内システムに共通する構造
- コードを書かない貢献
- AIエージェントによって参加のハードルが下がったこと
- AIを使っても、人間が判断すべきこと

「OSSへの貢献」という説明を、「ソフトウェア改善への参加」に置き換える。

#### 第2章　困りごとを変更要求に変える

- 「使いにくい」を具体化する
- 現在の挙動
- 期待する挙動
- 業務上なぜ問題なのか
- 影響を受ける人
- 発生頻度
- 例外条件
- 完了条件

社内システムでは、単なるバグだけでなく「業務ルールと実装が合っていない」というケースが多くなる。

#### 第3章　問題を共有し、着手してよいか確認する

- Issueやチケットを書く
- 社内チャットで相談する
- 開発チームや管理者を特定する
- 既存の要望や計画を確認する
- 実装前に相談すべき変更
- 自分で直してよい範囲
- 承認が必要な範囲

OSSではIssue、社内ではJira、Backlog、GitHub Issues、Slackなどを想定する。

### 第II部　AIとシステムを理解する

#### 第4章　そのソフトウェアがどのように管理されているかを知る

- ソースコードの所在
- ドキュメント
- Issue管理
- 開発環境
- テスト環境
- 本番環境
- 管理責任者
- 変更の承認フロー
- OSSにおけるCONTRIBUTING
- 社内における開発ルール

社内システムでは、リポジトリにアクセスできるからといって変更してよいとは限らない。

#### 第5章　安全な作業環境を準備する

- 本番環境を直接触らない
- 開発環境、検証環境、本番環境
- リポジトリを取得する
- 作業用ブランチを作る
- ローカルで起動する
- テストデータを用意する
- 個人情報や機密情報を使わない
- AIに渡してよい情報を確認する

OSS版では開発環境構築が中心だったが、社内版では環境分離と情報管理が重要になる。

#### 第6章　AIエージェントにシステムを調査させる

- 最初から修正させない
- 関係する画面、処理、ファイルを探す
- データの入力から出力までをたどる
- 業務ルールがどこに実装されているか調べる
- 類似機能を探す
- 過去の変更を調べる
- 調査結果に根拠を付けさせる
- 不明な点は担当者へ質問する

社内ではコードだけでなく、設定ファイル、データベース、外部サービス、運用手順が挙動に関係する場合がある。

### 第III部　小さな改善を作る

#### 第7章　変更を小さく設計する

- 一つの問題に絞る
- 変更範囲を限定する
- 業務への影響範囲を考える
- 既存の運用を壊さない
- コード変更以外の解決策を検討する
- 設定変更、手順変更、文言修正との比較
- 元に戻せる変更にする

社内システムでは、「コードを直すこと」が必ずしも最適解ではない。例えば次の選択肢がある。

- マニュアルを直す
- 入力チェックを追加する
- 初期値を変更する
- 設定を変更する
- データを補正する
- 業務フローを変更する
- コードを変更する

#### 第8章　AIに変更案を作らせる

- 目的、制約、完了条件を渡す
- 調査と実装を分ける
- 変更計画を先に作らせる
- 変更してはいけない部分を伝える
- 小さな差分を作らせる
- 差分を説明させる
- 不明点を推測させない
- 自分で説明できない変更を採用しない

#### 第9章　動作確認と受け入れ確認

- ビルドが通ること
- テストが通ること
- 業務上正しく動くこと
- 修正前後を比較する
- 正常系、異常系、境界条件
- 既存データへの影響
- 他部署や他ユーザへの影響
- 操作手順やマニュアルへの影響
- 現場の利用者による受け入れ確認

OSSの場合はPRを提出するが、社内では「受け入れ確認」が特に重要になる。

### 第IV部　安全に変更を届ける

#### 第10章　AIが作った変更をレビューする

- 要求と関係のない変更
- データ削除
- 権限変更
- 個人情報や機密情報の扱い
- 外部通信
- ログへの情報出力
- 依存ライブラリの追加
- パフォーマンスへの影響
- 障害時の挙動
- 元に戻せるか
- 自分だけでは判断できない変更

#### 第11章　エンジニアや管理者に変更を渡す

- Pull Request
- 社内レビュー
- 変更申請
- リリース依頼
- 問題、変更内容、確認結果を書く
- 業務上の背景を説明する
- 技術的な判断をエンジニアに委ねる
- レビューコメントへ対応する
- 自分でマージやリリースをしてよいか確認する

「メンテナとのコミュニケーション」を、より広い「責任者との協働」に置き換える。

#### 第12章　継続的に改善へ参加する

- 問題報告
- 仕様整理
- 受け入れテスト
- ドキュメント
- データ整備
- 小さな自動化
- ログや利用状況の分析
- 定期的な改善会
- ドメインエキスパートとエンジニアの役割分担
- 自分が詳しい領域を持つ
- 必要な技術を少しずつ学ぶ

## 4人の分担

前回の分担をほぼそのまま使う。

| 担当 | 担当範囲 | 中心テーマ |
| --- | --- | --- |
| A | 第1〜3章 | 現場の問題を要求やチケットに変える |
| B | 第4〜6章 | 管理体制、環境、コードやシステムの調査 |
| C | 第7〜9章 | 変更設計、AIによる実装、動作確認 |
| D | 第10〜12章＋全体編集 | 安全性、レビュー、組織との協働 |

社内システムまで含めることでDの担当内容が重くなるため、付録を次のように分担する。

| 担当 | 付録 |
| --- | --- |
| A | 問題整理シート、チケット作成テンプレート |
| B | システム調査プロンプト、開発環境確認リスト |
| C | 実装依頼プロンプト、受け入れテスト表 |
| D | 安全確認表、変更提出テンプレート、用語統一 |
